# USDC-USDT 方案可行性判断

## 已核实的客观事实

以下结论基于 2026-03-08 北京时间，本地直接拉取的 OKX 官方公开接口：

- `USDC-USDT` 现货 `tickSz = 0.0001`
- `USDC-USDT` 现货 `lotSz = 0.000001`
- `USDC-USDT` 现货 `minSz = 1`
- `USDC-USDT` 现货 `maxMktAmt = 1000000`
- `USDC-USDT` 现货 `maxLmtAmt = 20000000`
- 当前抓到的 `books` 快照里：
  - 卖一 `1.0000` 挂量约 `2.184M`
  - 买一 `0.9999` 挂量约 `7.720M`
- `USDC-USDT-SWAP` 当前仍然是 live，且资金费率接口返回 `fundingRate = 0`

## 对“10万U 赚 1U”的判断

- 这部分逻辑是成立的
- 因为 `tickSz = 0.0001`，所以 `100000 × 0.0001 = 10`
- 换句话说，`10万U` 如果满额完整吃到 `1 tick`，毛利就是 `10U`
- 想让 `10万U` 只赚 `1U`，对应价格变化要到 `0.00001`，这低于最小跳价

## 但为什么“有 1 tick 波动”不等于“你能稳定赚到 1 tick”

我本地用 OKX 官方 `history-candles` 拉了最近 `11000` 根 `1m` K 线，统计结果是：

- 至少 `1 tick` 的分钟：`10398 / 11000 ≈ 94.53%`
- 至少 `2 tick` 的分钟：`46 / 11000 ≈ 0.42%`
- 相邻分钟收盘变化：`4834 / 10999 ≈ 43.95%`

这只能证明“价格经常动”，不能证明“你的挂单经常成交且能完整回转”。

## 真正的摩擦点

### 1. 队列优先级

当前快照中：

- 买一总量约 `7.72M`
- 卖一总量约 `2.18M`

所以你的 `100k` 在规模上不算大，但在时间优先队列里未必靠前。

### 2. 库存偏移

如果买一先成交、卖一回补没成交，库存会持续偏向 USDC；反过来也一样。

### 3. 断线风险

如果公共行情断流，但旧挂单仍留在盘口，策略判断就会失真，所以需要重连后立即撤掉机器人托管挂单。

## 为什么 bot6 采用“REST 对账 + WS 行情 + WS 私有回报”

- `bot5` 已证明“执行器 + 私有订单回报 + 风控拦截”这条链路是对的
- OKX 官方文档明确写了 `orders` 私有频道不会推送订阅前已有挂单的初始快照
- 所以 `bot6` 启动时必须先调 `orders-pending`
- 下单继续先走 REST，排障更直接
- 行情继续走 WS，减少轮询延迟

## 为什么不把 swap 腿放进第一版

虽然 `USDC-USDT-SWAP` 当前资金费率确实是 `0`，但第一版仍不建议引入合约腿，原因是：

- 没有额外的明确边际收益证据
- 会引入保证金、仓位模式、合约单位、双腿时序问题
- 会把“做市问题”变成“做市 + 对冲执行问题”

## bot6 的结论

最终可行方案不是“看到 1 tick 就冲”，而是：

- 只做 `USDC-USDT` 现货
- 默认一档、一边一个挂单
- 默认 `shadow` 模式先观察
- 启动先 REST 对账，再开 WS
- 下单只用 `post_only`
- 库存超过 `40/60` 或 `60/40` 时切成单边修复
- 盘口陈旧、重连频繁、当日回撤过大时直接停机

这套方案是可行的，但它本质上是“低频稳健版稳定币微做市”，不是“几乎每分钟稳定白捡 10U”。
